Plan or Strategy?

Classify each statement as a Test Plan item (P) or a Test Strategy item (S). Wrong answers show an explanation.

Test Strategy = org-wide, high-level, stable — defines HOW the org approaches testing across all projects.
Test Plan = project-specific, detailed, dynamic — defines WHAT will be tested, by WHOM, and WHEN for a specific release.
"Our QA organisation follows a risk-based testing approach for all software products we develop."
"Ana will test the Login and Registration modules from April 5 to April 10."
"No release goes live until all Critical and High severity defects are closed or formally accepted by the Product Owner."
"Sprint 22 testing includes 45 automated regression tests and 20 manual exploratory tests for the Checkout redesign."
"All web applications we build must be validated against the OWASP Top 10 security vulnerabilities."
"Out of scope for this release: the Admin Portal, Email Campaign Management, and the Reporting Dashboard."
"The QA team applies a four-level severity classification (Critical / High / Medium / Low) across all projects in the organisation."
"Bram will execute performance testing for the Checkout redesign using JMeter before the December 15 release."

Severity vs Priority

Assign Severity and Priority to each defect. They are NOT the same thing.

Severity = how bad is the defect's impact on the system?  |  Priority = how urgently must it be fixed? (business need)
A low-severity bug can have high priority, and vice versa.
Defect Severity Priority
"Place Order" button returns a 500 error for all users during checkout.
The company logo has incorrect alt text ("img" instead of "BrandName logo").
The Privacy Policy link is broken on a low-traffic help page visited by ~10 users/month.
The admin dashboard crashes on load, but a simple URL workaround lets admins access all data.
Payment page shows currency as "USD" instead of "$" — the CEO is presenting a demo tomorrow.
Security scan reveals user passwords are stored in plain text in the database (invisible to end users currently).

Test Type Classification

Match each scenario to the correct test type.

After a new build is deployed, you open the app, navigate to the homepage, click login, and check that 3 core flows work — before running the full suite.
A developer fixes a bug in the login flow. You re-run every test in the test suite to make sure the fix didn't break anything else.
Without a test script, you navigate through the new payment flow trying unexpected inputs and edge cases to find bugs that weren't anticipated.
You use JMeter to simulate 5,000 concurrent users placing orders to check whether the system stays stable and meets the 2-second response SLA.
You attempt to access /admin/delete-user as a regular logged-in user by typing it directly in the URL bar — checking whether access controls are correctly enforced.
You recruit 5 participants unfamiliar with the product and ask them to complete registration while you observe — noting where they hesitate, get confused, or make errors.

Write a Bug Report

Fill in all sections of a professional bug report for the scenario below.

Scenario: You are testing an e-commerce app. When a logged-in user filters products by price range (e.g. $10–$50) and then clicks "Add to Cart" on a discounted item, the cart adds the item at its original full price — not the filtered/discounted price. This happens on Chrome (desktop) in the QA environment.
Bug Title — one clear, specific sentence summarizing the defect.
Steps to Reproduce — numbered, exact steps to recreate the bug.
Expected Behaviour — what should happen?
Actual Behaviour — what actually happens?
Severity and Priority — with justification.
Environment & Additional Info — browser, OS, build version, reproducibility.
